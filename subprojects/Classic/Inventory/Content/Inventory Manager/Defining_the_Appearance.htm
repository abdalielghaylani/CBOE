Defining the Appearance of Label

The appearance of the actual label/barcode is defined by creating or editing text files on the middle tier web server that include the codes necessary to create a useful barcode.

The text files can be created/edited using the ZPL programming language, or any number of other text-based languages that your printer supports.

This document is not intended to be a reference for any of these third-party languages. You can consult the following Zebra Web site for further details on how to set up the text files for a particular printer configuration:

http://www.zebra.com/id/zebra/na/en/index/products/special_features/proglang.html

 

The codes themselves reside in text files, and the Inventory application is capable of reading these files and inserting custom information where appropriate.

Each label type, such as containers, plates, or locations should have its own set of files that define how that particular label should look. These three types of labels are treated as just an “object”. The three files together constitute one record in the new “inv_lpr_definition” table, which is named “LPR Definitions” within the Manage Tables GUI. This GUI exposes the following fields:

• A header file [optional]
• A label file [required]
• A footer file [optional]

The purpose of including header and footer files is to provide the ability to issue a banner label that will print immediately before (and after) the string of barcode labels. This may be useful in cases where the label itself is nothing more than a barcode that does not contain human-readable information. For instances where the label itself is sufficient to identify what object it belongs to, no banner may be desired.

When managing the LPR Definitions table, you must provide the full path to the three text files on the middle tier web server. This path must also be accessible to the anonymous account with which IIS can be run. To ensure that the files are accessible, put them at the following location:

<webroot>\ChemOffice\ChemInv\config\xml_templates\LPR

The files themselves must be ASCII text files, and should represent as templates for building a full print job that will be sent to the printer. In order to print object-specific data on the label itself, the application has been designed to look for and replace custom tags inside the text body of each file with the relevant object data. These tags are context-sensitive, with respect to the type of label. The tags depend on whether the label is for plates, containers, or locations.

The underlying mechanism for selecting object data from the database for printing a label is identical to that used for the object reports. A SQL statement is executed that returns a result set with a large number of columns to it. The names of the columns are same as the ones that will be used to search for matching tags. In order to provide some level of uniqueness to each tag, and to avoid potential conflicts, the tag itself is surrounded by dollar signs ($TAG$).

To give an example, the following text file with ZPL codes might be used to print a barcode:

^XA

^LH0,50

^BX10,10,100^FD$BARCODE$^FS

^LH200,50

^FO^AO,40^FD$BARCODE$^FS

^LH200,100

^FO^AO,40^FD$SUBSTANCE_NAME$^FS

^PQ$NUM_COPIES$

^XZ

 

In this file, there are three custom tags that the application will replace:

• $BARCODE$
• $SUBSTANCE_NAME$
• $NUM_COPIES$

If the recordset had a barcode value of ‘abc’ and a substance name of ‘Acetone’, after processing the data being sent to the printer would look like the following:

^XA

^LH0,50

^BX10,10,100^FDabc^FS

^LH200,50

^FO^AO,40^FDabc^FS

^LH200,100

^FO^AO,40^FDAcetone^FS

^PQ1

^XZ

The only application-specific tag utilized by the system is the $NUM_COPIES$ tag, which is replaced by the value entered by the user during the printing process. In this sample ZPL, the ^PQ code is used to inform the printer how many copies to print. All other fields come from the result set columns. A full list of these columns for any given object can be viewed by opening the appropriate report in Access in Design View, and clicking on the “Field List” toolbar button. These reports are the following:

 

• rptContainerBarCodeLabels1 for containers
• rptLocationBarcodeLabels1 for locations
• rptPlateBarcodeLabels1 for plates
