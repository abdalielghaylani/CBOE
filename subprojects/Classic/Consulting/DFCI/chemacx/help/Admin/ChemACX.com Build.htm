Overview:

 

This document describes the process to set up the ChemACX.Com Enterprise Application for use as the public ChemACX.Com server.  Starting with version 9.0a2 of ChemACX Enterprise, all the .Com specific changes have been refactored and parameterized so that the public server can be easily configured without requiring the usual code hacks.  Note that ChemACX 9.0 is backwards compatible with COWS 8.1 so it can be setup under either COWS 2004 or COWS 2005 servers.

 

Installation:

 

-          Install, configure and test COWS CORE 2004 or 2005

-          Install ChemACX 9.0a2 or later (This can be done from the installer, or by manually checking-out the ChemACX project from SS and manually setting up the ChemACX virtual)

-          Set up IIS such that the ChemACX application is on its own website (This may require manually recreating ChemOffice, cfserverasp, cfwtemp and ChemACX virtuals)

-          During testing of the server upgrade, the website must be setup with a host header in the cambridgesoft.com domain to ensure that authentication cookies are shared with the accounts.cambridgesoft.com server.

-          Setup up a redirect.asp page as the default page for the web site and make the page Response.Redirect to “/chemacx/index.asp”

-          Copy the ChemACX database to the server

-          If a previous version of ChemACX existed on the server, clean up the cfwtemp cache to remove old cdx files.

-          Test the ChemACX application using the standard settings for ChemACX Enterprise

-          Create a folder named “uploads” under the ChemACX virtual to hold uploaded catalogs

 

Configuration:

 

The following files should be edited to configure the application for public service.  Only ini entries requiring a change are listed.  Other entries can be left in their default state.

 

global.asa:

-          Set the bUseCSWebUserAccounts variable near the top of the file to true

 

acxConfig.ini: 

[MSDX]

SHOW_MSDX_LOOKUP_LINK= 0

 

[CHEMREG]

REG_SERVER_NAME=NULL

REGISTER_FROM_ACX=0

 

[SHOPPINGCART]

EXPORT_TO_EXCEL= "False"

EXPORT_TO_WORD= "False"

EXPORT_TO_CHEMINV="False"

ADD_WITHOUT_PRICE= "True"

SHOW_LISTPRICE= "False"

SAVE_IN_DB= "False"

 

[SEARCHLOG]

LOG_SEARCHES= "True"

           

cfserver.ini:

[GLOBALS]

ALLOW_FLAT_SDFILE_EXPORT=0

ALLOW_HITLIST_MANAGEMENT=0

EXPIRE_HITLIST_HISTORY_DAYS=1

ALLOW_PUBLIC_HITLISTS=0

ALLOW_HITILIST_MNGMNT_FG_SELECTOR=0

ALLOW_QUERY_MANAGEMENT=1

EXPIRE_QUERY_HISTORY_DAYS=1

ALLOW_PUBLIC_QUERIES=0 

 

[CS_SECURITY]

USE_CS_SECURITY_APP=0

                                                                                                                                                                                                                                                           

chemacx,ini:

 

Limit the number of records exportable via sdf by adjusting all occurrences of MAXRECORDS:500 to a lower number like 100.

 

searchlog.udl:

 

-          Use SQL Server provider

-          Server Name: 172.17.1.18

-          Use SQL authentication

-          Username: cows2000

-          Password: cambridgesoft

-          Database:  xxxxxxxx (ask if you don’t know it)

-          Click to store the password in the udl

                                                                                                           

Maintenance:

Note that it may be necessary to turnoff Query management as it may consume too much database resources.  This has not been tested on the public server.  Definitely turn off hitlist management!  TODO:  Allow Pro users but disallow Net users to access Query and Hitlist Management.  This should be monitored once in production.
