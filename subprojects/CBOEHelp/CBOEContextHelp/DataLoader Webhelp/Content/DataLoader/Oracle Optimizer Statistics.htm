Oracle Optimizer Statistics

As always, Oracle performance is highly dependent on maintaining up-to-data statistics on all database objects in order to allow the Oracle query optimizer to chose the most efficient execution plan. During bulk load operations, the state of the database objects will change significantly as the database starts in a nearly empty state and ends with substantial volume of data in many tables. It is therefore critical to periodically gather optimizer statistics throughout the bulk load process.

Gathering Oracle Optimizer statistics is typically the purview of the Oracle DBA. There are several different approaches to do so. Below is a very simple script that gathers statistics for all the objects that may have an effect in the performance of Registration bulk loads. Consult a professional DBA for alternate methods that may be more appropriate for the particular database configuration. For example, on very large databases it may be preferable to gather statistics on a small percentage of rows rather than on all rows.

Connect system/manager@&&YourTNSName
Begin
dbms_stats.gather_schema_stats(ownname => 'REGDB', cascade => TRUE);
dbms_stats.gather_schema_stats(ownname => 'COEDB', cascade => TRUE);
dbms_stats.gather_schema_stats(ownname => 'CSCARTRIDGE', cascade => TRUE);
dbms_stats.gather_schema_stats(ownname => 'CHEMINVDB2', cascade => TRUE);
end;

Recommendations:

  • Breakup files to be loaded into manageable chunks of 10-50K records
  • Gather Optimizer Statistics between after loading some of the chucks
  • Flush the SQL shared pool after gathering statistics

connect system/manager@&&YourTNSName
alter system flush shared pool;
